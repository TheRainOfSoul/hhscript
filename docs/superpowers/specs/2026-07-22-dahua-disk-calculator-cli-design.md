# Клон Dahua Disk & Bandwidth Calculator (Basic Mode) в CLI

Дата: 2026-07-22
Статус: дизайн согласован, ждёт реализации

## Задача

Заменить приблизительный `Show-StorageCalc` в `menu.ps1` на точный клон
базового режима официального калькулятора Dahua
(`https://dhtools.dahuasecurity.com/DiskBandwidthCalculator/#/basic`).

Текущая реализация считает по выдуманной модели: фиксированный коэффициент
`0.476875` ГБ на (Mbps·ч), кодек как грубый делитель (H.265 = ÷2, H.265+ = ÷4).
Числа расходятся с официальным калькулятором.

Формулы для клона извлечены из JS-бандлов SPA (`js/chunk-120ececa.*.js`,
`js/app.*.js`), а не подобраны на глаз.

## Область работ

Клонируется **только Basic Mode** и **только одна группа камер** (на сайте
там таблица из нескольких строк с разными параметрами).

Не входит в работу:

- Advanced Mode (профили камер, субпотоки, хранение снимков, недельные
  расписания записи, вкладка результатов);
- несколько групп каналов с разными параметрами в одном расчёте;
- сохранение конфигурации между запусками;
- автотесты (сверка результатов — ручная, по сайту).

Удаляется: вопрос «запись по движению, % активности» — поля с такой семантикой
у Dahua нет, и он смешивал официальную цифру с оценочной.

## Извлечённые формулы

### Таблица базового битрейта по разрешению (Kbps)

| Разрешение | База | Разрешение | База |
|---|---|---|---|
| D1 | 1024 | 5MP | 6144 |
| 720P | 2048 | 6MP | 6144 |
| 960P | 2048 | 8MP | 8192 |
| 1080P | 4096 | 9MP | 8192 |
| 3MP | 4096 | 12MP | 12288 |
| 4MP | 5120 | | |

### Таблица минимального битрейта (Kbps)

| Разрешение | Мин. | Разрешение | Мин. |
|---|---|---|---|
| D1 | 128 | 5MP | 1024 |
| 720P | 384 | 6MP | 1152 |
| 960P | 384 | 8MP | 1536 |
| 1080P | 384 | 9MP | 1536 |
| 3MP | 640 | 12MP | 2304 |
| 4MP | 768 | | |

Используется только как порог предупреждения при ручном вводе битрейта.

### Расчёт битрейта (`calBitRate`)

Вход: разрешение `R`, кадры в секунду `s`, стандарт (PAL/NTSC),
кодек, environment `r`.

```
N = 25 (PAL) | 30 (NTSC)                 # опорная частота стандарта
s = min(s, N) только для D1/VGA/CIF      # прочие разрешения не клампятся
u = 0.8 ^ (1 / (N - 1))
m = base[R] / 6.4                        # 6.4 = 8 * 0.8
f = 20 * m / (9 + 2 * N)
g = f / 10

если 25 <= s <= 30:
    b = f/2 + g*N - g/2
    m = 8 * b
    d = 0.8                              # == u^(N-1)
иначе:
    b = f/2 + g*s - g/2
    m = 8 * b
    d = u ^ (s - 1)

m = m * d

кодек:  H.264       -> без изменений
        H.265       -> m / 2
        SmartH.264+ -> m * r
        SmartH.265+ -> m / 2 * r

m = округление ВВЕРХ до кратного 256      # (m + 255) & ~255 в оригинале
```

Environment: Low = 0.25, Medium = 0.5, High = 1.0. Влияет только на Smart+.

Контрольная точка: 1080P, PAL, 25 fps, H.264 → ровно **4096 Kbps**
(совпадает с базой таблицы).

### Объём и срок хранения

```
storageKB = битрейт_Kbps * часов_в_сутки * дней * 3600 / 8 / 0.9
дней      = floor(диск_KB * 0.9 * 8 / битрейт_Kbps / 3600 / часов_в_сутки)
bandwidth = каналы * битрейт_Kbps        # Kbps
```

Делитель `0.9` — заложенный Dahua запас в 10 %.

### Формат единиц

Пересчёт по **1024** (не 1000). Цепочка: KB → MB → GB → TB → PB → EB,
для полосы — Kbps → Mbps → Gbps → Tbps → Pbps → Ebps.
Делим, пока значение больше 1024 (максимум до EB), затем округляем
**вверх до одного знака после запятой**: `ceil(v * 10) / 10`.

## Архитектура

Всё живёт inline в `menu.ps1` — калькулятор обязан работать без сети
(скрипт уже загружен в память, а считают его обычно на объекте).
Движок отделён от диалога: чистые функции не вызывают `Read-Host`
и проверяются подстановкой значений.

### Чистые функции движка

| Функция | Вход | Выход |
|---|---|---|
| `Get-DahuaBitRate` | `-Resolution -FrameRate -VideoStandard -Compression -Environment` | битрейт, Kbps (кратен 256) |
| `Get-DahuaStorageKB` | `-BitRateKbps -HoursPerDay -Days` | объём, KB |
| `Get-DahuaDays` | `-BitRateKbps -HoursPerDay -DiskKB` | целое число суток |
| `Format-DahuaSize` | `-KB` либо `-Kbps` | строка вида `686.7 GB` / `32.8 Mbps` |

Таблицы `base` и `min` — два хеша на уровне скрипта, рядом с функциями.

### Диалог `Show-StorageCalc`

Порядок вопросов, у каждого дефолт по Enter:

1. Кол-во каналов — 1..1024, дефолт 1
2. Стандарт — PAL / NTSC, дефолт PAL
3. Разрешение — D1 / 720P / 960P / 1080P / 3MP / 4MP / 5MP / 6MP / 8MP / 9MP / 12MP, дефолт 1080P
4. Кодек — H.264 / H.265 / SmartH.264+ / SmartH.265+, дефолт H.265
5. Environment — Low / Medium / High, дефолт Medium. **Спрашивается только для Smart+**, потому что на прочие кодеки не влияет
6. FPS — 1..60, дефолт равен опорной частоте стандарта (25 или 30)
7. Битрейт — показывается рассчитанный, Enter принимает его, ввод числа перебивает
8. Часов записи в сутки — 1..24, дефолт 24
9. Что считать — `[1]` объём диска на N дней (дефолт), `[2]` срок из объёма диска.
   Для `[1]` спрашивается кол-во дней (дефолт 30);
   для `[2]` — объём и единица (ТБ по умолчанию, допускается ГБ)

### Вывод

Подписи те же, что в оригинале, чтобы результат можно было сверить глазами:

```
Channels             12
Bandwidth            32.8 Mbps
Required Disk Space  686.7 GB
```

либо, для обратного расчёта, `Storage Time  44 Days`.

Ниже — строка с параметрами расчёта (разрешение, кодек, fps, битрейт),
чтобы было видно, из чего получилось число.

## Обработка ошибок

- Значения вне диапазона: сообщение и повторный запрос, без падения.
- Нечисловой ввод: сообщение и подстановка дефолта — как в существующем
  `Read-Number`, его и переиспользуем.
- Ручной битрейт ниже минимума для выбранного разрешения: предупреждение
  жёлтым, расчёт продолжается (Dahua тоже не запрещает, а подсказывает).
- Нулевой битрейт или нулевые часы: `Get-DahuaDays` возвращает 0, диалог
  сообщает о некорректных данных и выходит.

## Проверка

Автотестов нет по решению заказчика. После реализации прогоняются
несколько наборов параметров, числа показываются пользователю, он сверяет
их с сайтом Dahua. Контрольные наборы для сверки:

1. 1080P, PAL, 25 fps, H.264 → битрейт 4096 Kbps
2. 4MP, PAL, 25 fps, H.265, 12 каналов, 24 ч, 30 дней
3. 8MP, NTSC, 30 fps, SmartH.265+, Medium, 4 канала, 12 ч, диск 4 ТБ → срок

`menu.ps1` дополнительно проходит существующий CI: парсинг и PSScriptAnalyzer.
